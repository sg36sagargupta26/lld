Design a Rate Limiter in Spring Boot

there are two major ways to implement rate limiter in Spring boot, that is either by using handlerinterceptor or filter
Lets create a very simple and naive rate limiter we are using concurrent hashmap and we have created a scheduled task which clears entire map every minute

We set say for the details(say ip/customerID) as key to map and when the customer say hits our api we trigger it and from there we are using map.merge to get current counts
and if its more than certain limit we throw 429 status too many requests

Q2> Why  are you using ZSET specifically?
Why not:
	•	List?
	•	Sorted list in app memory?
	•	Redis List?
A> Zset stores data sorted by score, and it allows for faster look up (O(log N)) compared to redis list which will take O(N)
Sorted List in app memory is okay if the volumes are small and single instance is running at a time, but when the volumes increase it impacts the program
slowing it down, also it can't handle multiple instance if two request hit two different instance than it will bypass the rate limiting
ZSET is preferred because it efficiently supports fast checking and deletion, which is critical for faster retrieal and checking the data
Q> Follow up: If I force you to use Redis List, what operation becomes inefficient and why?
The major one will be finding all the operation before the window start, and deleting them it can not efficiently find them

Q> Follow up: Why do we still prefer Lua script over MULTI/EXEC for this use case?
A> the problem with multi/Exec is that it can not use intermediate result it needs to do multiple network round trips
the advantage of lua script is it allows to use intermediate result and it condenses all the operation s in a single operations
so it only needs a single network trip
Q>Follow up: If you did try using WATCH + MULTI: What exactly would happen under high concurrency?
A> WATCH + MULTI uses optimistic locking, so under high concurrency many transactions fail and need retries, leading to high latency and reduced throughput.
Lua avoids this by executing everything atomically in one step without retries.
Q> Follow up: Why exactly is Lua atomic in Redis?
Lua is atomic because redis is single threaded and it blocks all other client code executions while lua is running
Q> Follow up:  If your Lua script takes 200ms to execute, What happens to other Redis requests?
A> Since redis is single threaded, it will block other requests, freezing them for a while and increasing latency
if  other request stay in queue for too long redis will start throwing busy error

Q>  What are your bottlenecks in Redis approach in case 50K request/sec, 10k User and burst traffic?
A: there are multiple bottlenecks in case of 50k rps and 10k user, usually each request has a memory footprint approximately (20bytes(userid) + 12 bytes(value: timestamp + noise)+ 8bytes(score)) roughly
40 bytes, so approx 40*50k*60 ~ 150MB so approximately, as not all memory is free instantly we need a buffer.
So the major challenge for our redis will be to free memory, so we need a custom memory eviction, and also we need to enable lazy evictions so it slowly evicts
automatically
The other bottleneck is processing a request, we need a chip powerful enough process each request under(60s/50k) ~ 1.2 ms, so to be on safer side we need to be
under 1ms

How can you redesign system so that:

Q> We don’t store every request (timestamp), but still enforce rate limiting
A> For this we need to implement token bucket as token bucket only needs to store a token count and a timestamp. Current approach requires
storing every timestamp, which results in better burst handling to current approach
In terms of
    - accuracy : sliding window is way more accurate as its the exact solution, but token bucket is a very good approximation
    - memory usage: token bucket has very low memory as it needs only token count and timestamp, but sliding window needs all timestamps(size == max request)
    - burst handling: token bucket is specially designed to handle burst up to its buket capacity, sliding window enforces a very limit and burst are capped
      by window limit
